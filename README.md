## 一句话讲清楚

这是一个跑在你自己电脑上的科研选题流程 skill。它不会给你 30 个花哨的题目让你挑花眼。它做的是另一件事，把你现有的文献、数据、笔记、研究基础先吃透，再围绕你提的模糊兴趣跑完五维扫描、文献矩阵、好问题压力测试，最后交给你 3 个主推选题加 2 个备选，每个都带证据、反证、推翻条件和两周 pilot 计划。

判断哪条方向值得做，是你自己的事。skill 的工作是让你做这个判断时，手里握着真实材料。

## 它解决什么问题

讲个真实场景。

说真的，某个博士生读了三年的平台劳动文献，笔记做了 1.5 万字，手头还有 8000 条招聘数据。开题前三个月，AI 给了他 20 个题目，他越看越焦虑，因为他根本判断不了哪个靠谱。最后他挑了一个标题最好看的那个，写到一半发现数据不支持，换方向，又换，再换，毕业差点延期。

这件事我相信很多做社科、经管研究的人都经历过。怎么讲，比较普遍。

这个 skill 重点解决「怎么判断一个题目值不值得做」，你看，这就是差异。具体怎么解决，靠下面这几条硬规则。

**材料先于提问**。你必须先交文献、笔记或题目摘要，skill 才会启动。空喊「给我几个新颖的选题」，它不接。这条规则是为了把 AI 自己的背景知识挡在外面，逼着它用你的材料说话，我始终坚信这一点。

**每个判断都带反证**。你说某方向有价值，skill 必须同时给你这条方向的反面证据。论文 A 说算法管理影响收入，论文 B 说得分高低才是关键，论文 C 说样本本身就是错的。你得看到全貌再选。

**阶段冻结**。scope 一旦确定，后面不许偷偷改题。改就开新工作目录。这条规则是为了防止「写到一半换方向」这个最常见的翻车原因。

**独立审查**。关键节点（扫描、选题）必须由另一个审查人格重新过一遍，不能自审自盖合格章。

**敢于让你停下来**。很多方向其实根本不值得做。skill 会主动告诉你，这条方向继续、修改、暂缓还是淘汰。它不会为了显得有用而硬推荐。

我自己做这个 skill 的过程里，写代码反而简单，难的是说服自己接受一个事实。怎么说呢，skill 帮你思考，但替你思考这事儿它不擅长。最重要的判断，最终还得是你做的。

## 它和同类有什么不同

市面上做研究选题的工具大致分两类。一类是 AI 题目生成器，输出一堆标题加摘要，让你挑。另一类是文献综述工具，把一堆论文总结给你。

这个 skill 不属于任何一类。它是研究决策闸门，专注于选题过程的透明度。几个最硬的差异点。

**证据 + 反证 + 推翻条件 + pilot** 这四样默认就有。同类工具很少要求每个候选问题必须能证伪。

**多阶段 + 阶段闸门**，scope、materials、scan、topics、final 五道机器校验加两道人工独立审查。同类工具大多是一次性对话，没有阶段产物校验。

**可退出**。同类工具几乎不会主动告诉你「这个方向别做了」。skill 会。背后是协议在驱动，我其实挺喜欢这个设计。

**完全本地，无外部 API**。v1.6 之后不需要 Semantic Scholar，不需要 Gemini，不需要任何 API Key。MinerU 跑在你本机，PyMuPDF 跑在你本机，所有证据台账都在你工作目录下。隐私问题直接消失。

## v1.6 新增了什么

如果你用过 v1.5.2，会发现流程里多了三件事。说真的，都是被真实场景逼出来的改动。

**先摸底，再追问**。v1.5.2 直接问你「关心什么问题」。v1.6 先问你是谁、你从哪来、你要到哪去，学科背景、当前阶段、历史沿革、研究基础、交付目标、时间约束六个维度。你回答完之后，skill 才会基于这些具体信息追问，换种问法。

为什么这么改。其实空问得到的回答一定是模糊的，因为用户自己也没想清楚。但如果你告诉 skill 你之前做过平台劳动的收入研究、现在转向职业稳定性、博士二年级、Stata 玩得熟、有 8000 条招聘数据，skill 就能问出有意义的问题，比如「你说之前做过收入，为什么不做了」「没有骑手访谈数据，光靠招聘数据撑得住吗」。

**MinerU 批量提取全文**。v1.5.2 时，skill 只登记你的 PDF 文件，不读内容。Agent 做后续判断基本是凭标题和摘要猜。v1.6 跑了一个 MinerU 批量提取脚本，把你上传的 PDF 全转成 Markdown 全文，含表格和公式。MinerU 是你本机已经装好的命令行工具，没有额外依赖。万一 MinerU 挂了，脚本自动回退到 PyMuPDF，保证流程不阻塞。

提一句细节，扫描版 PDF（图片型）MinerU 和 PyMuPDF 都无能为力，这点我踩过坑。这种情况下脚本会标记为 failed 并跳过，不假装成功。处理扫描版需要你自己先 OCR，或者直接提供 Markdown 文本。

**文献矩阵**。v1.5.2 的五维扫描是叙述性的，可比较性弱。v1.6 在扫描之后新增了「研究问题 × 方法」二维矩阵，把每篇提取出来的论文按研究问题和方法归入对应单元格。空白格自动标注「待验证」，进入核心缺口判断。

为什么要做这个矩阵。说到底，叙述性的「领域分三支」很容易自欺欺人。把论文摆进二维表，空白格一眼可见。哪个组合没人做、哪个组合被做烂了，看表就懂。

局限要先说在前面，我自己也不藏着。矩阵归一化目前还是规则匹配，可能漏合并语义相近但表述差异大的标签。LLM 辅助归一化是后续版本的事。空白格也不必然等于有价值的研究缺口，它也可能是检索不全或归一化失败。所以矩阵只是辅助，Phase 6 的核心缺口判断还是要人来把关。

## 流程长什么样

```
摸底（六个维度）
  ↓
三问（基于摸底的具体追问）
  ↓
协议冻结 → scope 闸门
  ↓
材料登记 + 研判 → materials 闸门
  ↓
MinerU 批量提取 → papers 闸门 ★新增
  ↓
五维扫描（学术文献维度优先用提取的全文）
  ↓
问题域地图 + 文献矩阵 ★新增
  ↓
独立审查 → scan-review 闸门（含矩阵校验）
  ↓
文献脉络
  ↓
趋势判断
  ↓
核心缺口（输入含矩阵空白格、扫描发现、趋势转折）
  ↓
好问题压力测试 → questions 闸门
  ↓
3+2 选题 → topics 闸门（含独立审查）
  ↓
交付 → final 闸门
```

全程 7 道机器闸门 + 2 道独立人工审查。任何一道不过，下游就不启动。

## 快速上手

一行装上。

```bash
npx skills add jefeerzhang/research-topic-selection
```

或者手动克隆。

```bash
git clone https://github.com/jefeerzhang/research-topic-selection.git
# 复制到你的 Agent skills 目录
```

环境要求，已经装好的就不用管，没装的按提示补一下。

- Python 3.10+
- `mineru-open-api` 命令行（PDF 提取用，没有的话脚本会回退到 PyMuPDF）
- `pytest`（跑测试用）

装完对 Agent 说一句。

```text
我想用 research-topic-selection 来确定一个研究选题。我的主题是「平台劳动者的算法管理与职业稳定性」。

我手头的材料
平台劳动文献笔记 Markdown 约 1.5 万字
一份问卷设计经验（之前做过非随机抽样）

公开招聘数据 某平台 2022-2024 年的岗位描述

请用标准研究构想作为交付物，学科分支为实证量化，时间窗 2022-2026，语言中文。
```

Agent 会先问你六个摸底维度，然后启动流程。

也可以直接说，「用 research-topic-selection 摸一下我的基本情况」，它就知道你想从哪开始。

## 能交付什么

启动之后，工作目录里会按编号出现 9 个 Markdown 文件加一份证据台账。

| 文件 | 内容 |
|---|---|
| `00_protocol.md` | 冻结的研究协议。题目、时间窗、学科分支、交付物、风险档位 |
| `01_三问与澄清.md` | 基于摸底的具体三问回答 |
| `02_用户材料研判.md` | 材料概要、用户已有研究问题、证据冲突、检索约束 |
| `user_materials/extracted/*.md` | MinerU 提取的论文全文 ★新增 |
| `03_五维扫描.md` | 政策 / 文献 / 实践 / 数据 / 发表窗口五个维度的扫描报告 |
| `03A_文献矩阵.md` | 研究问题 × 方法二维表 ★新增 |
| `04_问题域地图.md` | 6 点结构化的领域分析 |
| `05_文献脉络.md` | 前沿方向、核心争论、方法谱系 |
| `06_趋势判断.md` | 四维度趋势加阶段判断 |
| `07_核心缺口.md` | 1-3 个核心缺口，每个含「已解释/仍不足/为何重要」三段式 |
| `07A_好问题卡.md` | 排名最高的 1-3 个候选问题完整卡，含推翻条件和两周 pilot |
| `08_选题推荐.md` | 3 个主推 + 2 个备选，加推荐判断 |
| `review/delivery_manifest.json` | 全链路产物血缘 |

每个候选问题都会强制回答这几个字段。

| 字段 | 例子 |
|---|---|
| 研究价值 | 为什么算法管理对职业稳定性值得现在研究 |
| 竞争性解释 | 收入下降、技能贬值、身份认同是否也能解释同一现象 |
| 关键证据 | 平台规则变化、劳动者访谈、公开招聘数据 |
| 推翻条件 | 如果算法评分与离职率无显著相关，则该方向应被淘汰 |
| 两周 pilot | 先用招聘数据做描述性统计，再决定是否需要深入访谈 |
| 当前决策 | 继续 / 修改 / 暂缓 / 淘汰 |

每个字段都必须基于你提供的材料，或基于扫描后形成的证据台账。模板填空做不到这点。

## 安全边界

做这个 skill 的过程里我反复提醒自己几件事，写在这里让你也看到。

**不伪造来源**。MinerU 提取失败就标 failed，没有材料就停。不编造文献，不杜撰数据。这是写死的第一规则。

**不替代学术判断**。skill 给的是审计和决策支持，最终选题必须研究者本人确认。它不会替你下决心。

**不跳过关键阶段**。scope 没过不进扫描，scan 审查没过不出问题卡，topics 审查没过不交付。每个闸门都是硬约束，必须执行。

**不碰隐私数据**。所有材料你自行管理，skill 只读你给的本地路径，不上传任何东西。MinerU、PyMuPDF 都跑在你本机。

**不保证发表**。skill 提高的是选题过程的透明度，不承诺任何学术成果。它不背书你的题目一定能写出来、能发出来、能拿基金。
